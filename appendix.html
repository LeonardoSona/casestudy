Appendix

Medallion Mapping (Bronze → Silver → Gold)

  • Bronze (Raw/Landing): source-faithful ingests from applications/operational DBs; immutable for auditability.
  • Silver (Canonical Core = Source Data Products): quality-verified, standardized, conformed to enterprise keys; “general-purpose” data products designed for broad reuse.
  • Gold (Curated/Product = Consumer Data Products): on-demand, outcome-specific products composed from multiple Source Data Products; may be materialized or virtual (via a fabric/virtualization layer).

Glossary tie-in: Source systems → Bronze → Source Data Products (Silver). Interoperable composition (mesh/fabric) → Consumer Data Products (Gold), delivered to data consumers.

Source vs Consumer Data Products (in the Mesh/Fabric)

At the bottom of the diagram sit source systems—applications and operational databases that generate operational data.

Data from these systems is transferred into the data mesh and represented as Source Data Products. A Source Data Product can aggregate from one or many systems and, during onboarding, undergoes quality verification and standardization to produce a canonical representation aligned to enterprise keys and semantics. These “general-purpose” products support broad business needs.

In contrast, Consumer Data Products are created on demand for specific outcomes (e.g., a KPI, workflow, or decision). They are typically the result of transformations and joins across multiple Source Data Products—this is the heart of the mesh’s interoperability. A data fabric pattern can power this layer: consumer products may be materialized or virtual (a semantic/virtual management layer atop distributed sources).

Both Source and Consumer Data Products are cataloged and made discoverable with access controls, contracts, lineage, and SLAs.

  • Source Data Products = standardized, governed, reusable “general-purpose” building blocks.
  • Consumer Data Products = on-demand outcomes; can be materialized or virtual via fabric/virtualization.
  • Mesh ensures interoperability across products/platforms/clouds; fabric enables virtual composition.

Dual-Layer Data Products — Step-by-Step

1) Define keys & semantics (Source/Core): MATERIAL_ID, SITE_CODE, BOM_ID, INVENTORY_DATE.

2) Source (Canonical) schemas:

material_core(material_id, material_type, base_uom, global_status)
bom_core(material_id, bom_id, component_material_id, qty_per)
inventory_core(material_id, site_code, on_hand_qty, blocked_qty, inventory_date)

3) Domain Extension (joins back to Source):

qsc_batch_quality_ext(material_id, site_code, batch_id, manufacture_ts,
                      qa_release_ts, assay_result_pct, deviation_flag)

4) Contracts: Source = enterprise standard; Extension = domain-owned; must join on declared keys.

5) Catalog: Tag as source vs extension; show lineage and sample joins.

6) Usage: Enterprise = Source; QSC/Consumers = Source + Extensions/Compositions.

7) Change: Source = rare, major versions; Extensions/Consumer = faster, minor versions when additive.
