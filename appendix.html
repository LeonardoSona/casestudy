Appendix

Medallion Mapping (Bronze → Silver → Gold)

  • Bronze (Raw/Landing): immutable, source-faithful ingests for auditability.
  • Silver (Canonical Core): our Enterprise Core — standardized keys, cleansed & conformed, with contracts, lineage, SLAs and DQ rules.
  • Gold (Curated/Product): Shared Views (reusable slices), QSC Extensions (domain-specific detail that joins to Silver), and Cross-Domain Products (compositions for decisions).

Consumption pattern: Finance/Commercial can use Silver/Core as-is; QSC joins Gold-Domain Extensions to Silver for batch/QA detail.

Change policy: Silver/Core changes are rare & versioned (major if breaking); Gold extensions evolve faster (minor when additive).

Glossary tie-in: Core assets = Silver; Shared views = Gold-Shared; QSC extensions = Gold-Domain; Cross-domain products = Gold-Product.

Dual-Layer Data Products — Step-by-Step

1) Define keys & semantics (Core): MATERIAL_ID, SITE_CODE, BOM_ID, INVENTORY_DATE.

2) Core schemas:

material_core(material_id, material_type, base_uom, global_status)
bom_core(material_id, bom_id, component_material_id, qty_per)
inventory_core(material_id, site_code, on_hand_qty, blocked_qty, inventory_date)

3) QSC Extension (child only, no Core duplication):

qsc_batch_quality_ext(material_id, site_code, batch_id, manufacture_ts,
                    qa_release_ts, assay_result_pct, deviation_flag)

4) Contracts: Core = enterprise standard; Extension = QSC-owned; must join on declared keys.

5) Catalog: Tag as core vs extension; show lineage and sample joins.

6) Usage: Enterprise = Core; QSC = Core + Extension.

7) Change: Core = rare, major versions; Extension = faster, minor versions.
